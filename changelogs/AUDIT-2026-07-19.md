# Production Readiness Audit — S_milk — 2026-07-19

## Executive summary

**Verdict: NOT READY FOR PRODUCTION.**

The current working tree builds and all 28 automated tests pass, but the app has
release-blocking security, availability, and financial-integrity defects. The
most serious problems are:

1. The six-digit PIN is only a client-side UI gate; Supabase RLS allows anonymous
   users to read, modify, and delete all operational data.
2. Login and PIN rotation do not match the checked-in SQL RPC signatures.
3. Monthly bill generation writes zero-value bills.
4. Toggling a delivery resets its quantity to zero and clears its note.
5. Payment idempotency is checked after the bill is changed, so retries can
   double-count money.
6. Adjustments and credit notes are non-transactional and can partially or
   repeatedly alter bills.
7. Important normal-use flows—subscriptions, customer deactivation, historical
   delivery dates, import totals, and auto-rollover—are broken.

This audit covers the current on-disk working tree, including uncommitted and
staged changes. It supersedes stale findings in the July 15 reports where the
implementation has since changed.

## Scope and method

Reviewed:

- `src/lib/**`: Supabase client, API actions, mappers, utilities, filters,
  validation, and tests.
- `src/hooks/**`: auth, entity state, derived state, busy handling, and all
  domain handlers.
- `src/components/**` and `src/pages/**`: shell, pages, modals, forms,
  accessibility, mobile UX, and destructive actions.
- `public/**`, `index.html`, and `manifest.json`: service-worker registration,
  caching, update behavior, CSP, and installability.
- `supabase/**`: schema, constraints, triggers, RPCs, grants, and RLS.
- Tooling and operations: package scripts, CI, lint, tests, coverage, build,
  dependency audit, formatting, spellcheck, cycle detection, and documentation.

Severity scale:

| Severity | Meaning                                                                       |
| -------- | ----------------------------------------------------------------------------- |
| Critical | Exploitable access, core workflow outage, or direct financial/data corruption |
| High     | Normal-use bug, partial data loss, or major production control missing        |
| Medium   | Material UX, reliability, maintainability, or defense-in-depth weakness       |
| Low      | Hygiene, compatibility, or low-risk quality issue                             |

# Critical findings

## C1. The PIN does not protect any Supabase data

**Evidence**

- `supabase/rls_policies.sql:31-45` grants `FOR ALL` access to `anon` and
  `authenticated` on every business table with `USING (true) WITH CHECK (true)`.
- `supabase/rls_policies.sql:43-45` also allows anonymous reads and writes to
  `settings`, including the PIN hash and lockout fields.
- `src/hooks/useAuth.js:20-37` stores the token returned by `verify_pin` only in
  `sessionStorage`.
- `src/lib/supabaseClient.js:3-6` always creates an anonymous Supabase client.
- No API action sends the returned token to Supabase, validates it, installs it
  as a Supabase Auth session, or checks it in RLS.

**Impact**

Anyone who extracts the public Supabase URL and anon key from the browser bundle
can bypass the login screen and directly read, create, alter, or delete customer,
delivery, supplier, subscription, payment, and billing data.

**Required remediation**

- Use real Supabase Auth and an authenticated JWT for the operator.
- Disable public signup and restrict RLS to the intended operator identity or an
  explicit admin claim.
- Deny `anon` access to all business tables and `settings`.
- Expose only narrowly scoped RPCs for sensitive mutations.
- Do not put a service-role key in the browser.

## C2. Anonymous callers can erase the entire database

**Evidence**

- `src/lib/api.js:1227-1254` performs direct deletes across all business tables;
  the only guard is the browser-side string `payload.confirm === "ERASE"`.
- `supabase/security_functions.sql:139-152` exposes a `SECURITY DEFINER`
  `erase_all_data` RPC and grants execution to `anon`.
- The RPC also protects itself only with the public literal `ERASE`.
- PostgreSQL function execution is granted to `PUBLIC` by default unless
  explicitly revoked.

**Impact**

Any anon-key holder can permanently delete all operational data without knowing
the PIN.

**Required remediation**

Revoke execution from `PUBLIC` and `anon`, require a real authenticated operator
claim, require recent re-authentication, perform the operation in one audited
server-side transaction, and maintain tested backups and restore procedures.

## C3. Login and PIN rotation are broken, and lockout does not persist

**Evidence**

- `src/lib/api.js:262` calls `verify_pin` with `input_pin`, while
  `supabase/security_functions.sql:29` declares `p_pin`.
- `src/lib/api.js:1211-1214` calls `rotate_pin` with `current_pin` and `new_pin`,
  while `supabase/security_functions.sql:77` declares `p_current` and `p_new`.
- Supabase REST RPC arguments are resolved by their declared names.
- `supabase/security_functions.sql:68-69` updates `failed_attempts` and then
  raises an exception. The exception aborts the transaction and rolls back the
  lockout update.
- The PIN row is read without `FOR UPDATE`, so concurrent attempts race.
- `supabase/security_functions.sql:18-24` and `:45-47` pass arbitrary hex bytes
  to `crypt()` instead of a valid `gen_salt(...)` result.
- `supabase/base_schema.sql:159-160` seeds every new install with the known PIN
  `123456`.

**Impact**

A clean deployment from the checked-in SQL cannot reliably log in or rotate the
PIN. Brute-force attempts are not durably counted, and the default credential is
known.

**Required remediation**

- Make JS and SQL RPC argument names identical and contract-test them against a
  disposable Supabase database.
- Use `crypt(pin, gen_salt('bf'))` or a server-side password hashing service.
- Lock the settings row with `FOR UPDATE`.
- Return a failure result after updating counters instead of raising in the same
  transaction.
- Force PIN rotation during initial setup and never expose PIN/hash columns.

## C4. Toggling delivery status destroys delivery quantity and note

**Evidence**

- `src/hooks/handlers/useDeliveryHandlers.js:12-20` sends only `logId` and
  `delivered` to `updateLogEntry`.
- `src/lib/api.js:416-424` nevertheless writes `qty: toNum(payload.qty)` and
  `note: payload.note`.
- Missing `qty` becomes zero; missing `note` becomes null/undefined.

**Impact**

Every delivered/skipped toggle can set the delivery to `0 L` and clear its note.
This corrupts daily totals, inventory, and any billing derived from delivery
logs.

**Required remediation**

Make `updateLogEntry` a true partial update that only writes fields explicitly
provided, or pass and version-check the full current log. Add an integration test
that toggling preserves quantity, product, customer, date, and note.

## C5. Monthly bill generation writes zero-value bills

**Evidence**

- `src/hooks/handlers/useBillOperations.js:24-28` sends `ratePerLiter`.
- `src/lib/api.js:740-805` ignores `ratePerLiter`, reads `amount`, and stores
  `toNum(undefined)`, which becomes zero.
- Neither `customers` nor `subscriptions` has a mapped/schema rate field even
  though `useBillOperations` tries to read one.
- No delivered-log aggregation occurs.
- `src/pages/Dashboard.jsx:119-121` calls generation with no month, causing every
  request to fail `customerId and month are required`.

**Impact**

The core billing workflow creates or overwrites bills at ₹0. The dashboard entry
point fails entirely.

**Required remediation**

Calculate the bill from delivered logs and an authoritative effective rate in a
single transactional RPC. Define where rates live, map them consistently, pass
an explicit month from every entry point, and test nonzero quantities, locked
bills, retries, and concurrent generation.

## C6. Payment retries can double-count money and lose audit records

**Evidence**

- `src/lib/api.js:687-710` updates the bill before recording the payment
  idempotency key.
- `src/lib/api.js:713-729` inserts the payment record afterward and ignores a
  duplicate-key error after the bill has already been incremented again.
- Other payment-insert errors are only logged; the UI still reports success.
- The bill update and payment insert are separate HTTP/database transactions.
- `src/lib/api.js:217-225` maps metadata as `mode` and `date`, but
  `src/lib/api.js:644-652` reads `paymentMode` and `paymentDate`, so persisted
  metadata becomes `Unknown` and today.
- The existing SQL RPC at `supabase/security_functions.sql:104-136` is unused,
  does not insert a payment row, ignores mode/date/note, has no idempotency key,
  and does not reject non-positive amounts.

**Impact**

A retry after a timeout can charge the same payment twice. A bill can show money
received without a corresponding payment audit record, and recorded metadata can
be wrong.

**Required remediation**

Use one transactional RPC that claims/checks the idempotency key before any
mutation, locks the bill, validates a positive amount and remaining balance,
inserts the audit record, updates the bill/version, and returns the original
result for repeated keys.

# High findings

## H1. Adjustment and credit-note application are unsafe and currently broken

**Evidence**

- `src/lib/api.js:864-935` and `:1256-1329` mutate the bill and source record in
  separate transactions.
- Their version-filtered bill updates do not use `.select().single()`, so a
  zero-row compare-and-set update does not produce the `PGRST116` error those
  branches expect.
- `supabase/triggers.sql:18-19` assigns `NEW.updated_at` for adjustments and
  credit notes, but those columns do not exist in `supabase/base_schema.sql:113-136`.
  Updates to those tables therefore fail at runtime.
- Credit-note application explicitly writes the nonexistent `updated_at` column
  after changing the bill.
- `src/lib/api.js:131-140` maps an adjustment ID as `adjustmentId`, while
  `src/pages/More.jsx:100-125` reads `a.id`; Apply receives `undefined`.
- New adjustments contain no target bill, but `src/pages/More.jsx:122-126`
  passes `disableApply` to a component that ignores the prop.
- `applyAdjustment` marks the adjustment applied even when `billId` is absent.
- Loaded credit notes have no visible application workflow.

**Impact**

Money can be changed without the source record being marked, marked without a
bill change, or applied twice on retry. The current UI cannot reliably apply an
adjustment at all.

**Required remediation**

Normalize the mapper to `id`, require and validate a same-customer target bill,
add/remove the intended timestamp columns/triggers, and move each complete money
operation into one row-locking transaction.

## H2. Import totals shown to the user are not the totals saved

**Evidence**

- `src/components/modals/ImportModal.jsx:6-9` computes `qty * rate` for display.
- That value is never written to the form.
- `src/lib/api.js:198-205` persists `Number(form.total)`, which is zero/NaN for a
  new import and stale when editing quantity/rate.

**Impact**

Purchase-cost reports can be wrong even though the modal showed the correct
amount before saving.

**Required remediation**

Derive `total_cost` from normalized quantity and rate in the API/database, not a
separate editable form field. Test create and edit behavior.

## H3. Subscription add/edit controls throw and save incompatible fields

**Evidence**

- `src/hooks/useAppState.js:42-44` defines a setter that only accepts DOM events.
- `src/components/modals/SubscriptionModal.jsx:18-23` passes an array to it.
- `src/components/modals/SubscriptionModal.jsx:102-112` passes a boolean to it.
- `src/components/modals/SubscriptionModal.jsx:79-86` writes `quantity`, while
  `src/lib/api.js:1090-1121` reads `qty`.
- `src/components/AppModals.jsx:166-183` checks `ctx.setForm`, but `setForm` is
  not exposed by `useAppUi`; it then calls `openModal` without the subscription,
  resetting form state to `{}`.
- The add defaults use `quantity` and uppercase `FULL_CREAM`, which do not match
  the canonical mapper/constant shape.

**Impact**

Delivery-day and Active controls throw at runtime, edits lose their data,
quantity can save as zero, and an intended edit can become an insert.

**Required remediation**

Use an explicit `setField(name, value)` API, expose only the setter needed by
modals, pass the selected subscription into `openModal`, and standardize on
`qty`, canonical milk types, and complete initialized defaults.

## H4. Auto month rollover is incompatible with current schema/contracts

**Evidence**

- `src/lib/api.js:338-360` selects and updates `subscriptions.end_date`, which is
  absent from `supabase/base_schema.sql:100-111`.
- Supabase update/lookup/insert errors inside the loop are not checked.
- `src/App.jsx:30-36` reads `res.data.*`, but `callApi` returns `result.data`
  directly at `src/lib/api.js:1340`.
- `new Date(y, m, 0).toISOString()` can produce a previous calendar date in a
  positive-offset timezone.
- Rollover creates zero-value bills and is coordinated only through localStorage
  on one device.

**Impact**

Rollover fails, reports false success, or partially updates data. It can repeat
on every mount and generate zero-value bills.

**Required remediation**

Remove the feature until its schema exists, or implement a transactional,
idempotent database RPC with a server-recorded completion marker and explicit
Asia/Kolkata-safe month boundaries.

## H5. Selecting another delivery date cannot reliably show that date

**Evidence**

- `src/hooks/useEntityStore.js:76-83` replaces `logs` with the selected date.
- `src/hooks/useAppDerived.js:21-24` always filters that array against
  `getToday()`, not `state.logDate`.
- `src/pages/Delivery.jsx:103-107` can issue overlapping requests, but
  `fetchLogs` has no abort signal or request sequencing.

**Impact**

Past/future dates appear empty, and a slower stale response can overwrite the
most recently selected date.

**Required remediation**

Derive delivery-page logs using `logDate`, keep dashboard-today state separate,
and ignore/abort stale requests.

## H6. Customer deactivation is wired to the wrong handler

**Evidence**

- `src/components/AppPage.jsx:38-40` passes `handlers.updateCustomer` as
  `onDeactivate`.
- `src/pages/Customers.jsx:136-138` calls it with only the customer ID.
- `src/hooks/handlers/useCustomerHandlers.js:23-33` expects a complete form and
  version.
- A dedicated `deactivateCustomer` API action exists but is unused and itself
  bypasses optimistic concurrency.

**Impact**

The Deactivate button does not reliably deactivate the customer and may submit
an invalid empty update.

**Required remediation**

Add a dedicated version-aware handler receiving the customer object, require
confirmation, and perform a compare-and-set status update.

## H7. Concurrent daily generation can overwrite an existing delivery

**Evidence**

- `src/lib/api.js:458-476` reads existing logs separately from the write.
- `src/lib/api.js:546-552` uses normal upsert on conflicts.
- A log inserted or edited between those operations is updated by the upsert,
  including `delivered: false`, quantity, product, and note.

**Impact**

A concurrently created or already delivered record can be reset to pending and
have its details replaced.

**Required remediation**

Use `INSERT ... ON CONFLICT DO NOTHING`/`ignoreDuplicates` in a database-side
operation and return the actual inserted count.

## H8. Displayed form defaults are not submitted defaults

**Evidence**

- Modals display fallback values without initializing `form`.
- `PaymentModal.jsx:35-59` displays pending amount and today, but
  `useBillPayments.js:23-29` reads missing `form.payAmt` and rejects Save until
  the user edits the amount.
- `AdHocLogModal.jsx:25-31` displays `1`, but the API receives missing quantity
  if untouched.
- Similar visual-only defaults exist for new customer quantity and import/pause
  dates.

**Impact**

Users see valid values, press Save, and receive errors or persist zeros/missing
values.

**Required remediation**

Initialize complete form objects when opening each modal and also normalize
safe defaults at validation/API boundaries.

## H9. The SQL directory is not a deployable migration system

**Evidence**

- `supabase/*.sql` uses plain `CREATE TABLE`, `CREATE POLICY`, `CREATE TRIGGER`,
  and `ADD CONSTRAINT` statements.
- There is no `supabase/migrations/` history or tested deployment command.
- Applying the files to an existing project will collide with existing objects.
- The current SQL and JS already disagree on RPC arguments, columns, and money
  workflows.

**Impact**

A manual production update can fail halfway and leave a mixed schema, open RLS,
known default PIN, or incompatible frontend.

**Required remediation**

Create ordered versioned migrations, test clean install and upgrade from the
current production schema, wrap coherent changes in transactions, and add a
schema/RPC/RLS integration job to CI.

## H10. Core database invariants are not enforced

**Evidence**

- `supabase/base_schema.sql` lacks checks for positive/nonnegative quantities,
  payments, amounts, and versions.
- Customer foreign keys are nullable on several business tables.
- Status values, `YYYY-MM` month format, delivery-day JSON shape, pause date
  order, and `amount_paid <= amount` are not constrained.

**Impact**

Any direct or buggy client write can create invalid financial and scheduling
states that later code is not designed to handle.

**Required remediation**

Clean existing data, then add validated `NOT NULL`, foreign-key, status, date,
JSON, numeric, and version constraints.

## H11. CI and tests provide a false production signal

**Evidence**

- `.github/workflows/ci.yml:27-34` runs only lint, tests, and build.
- `package.json:22` claims a broader `check-all`, but that command currently
  fails at Fallow.
- Coverage is only **14.51% statements, 10.79% branches, 13.98% functions, and
  14.53% lines** with no thresholds.
- `src/lib/api.js` has only **6.95% line coverage**.
- Auth, entity loading, most pages/modals, subscriptions, rollover, and service
  worker behavior have no meaningful automated coverage.
- Handler mocks in `src/hooks/useAppHandlers.test.js:52-80` return a nested
  `{ success, data }` contract while production `callApi` returns data directly.
- The current two API tests cover only overpayment and one version conflict.

**Impact**

CI can be green while core operator flows and production SQL are broken.

**Required remediation**

Gate CI on a deterministic `check-all`, add per-file thresholds for financial
and auth code, assert exact API payloads/results, and add disposable-database
integration tests plus one full operator-flow E2E test.

## H12. The app is not operationally offline

**Evidence**

- `public/sw.js:123-142` correctly keeps Supabase network-only.
- There is no IndexedDB snapshot, mutation queue, sync status, or conflict
  resolution.
- `src/components/AppShell.jsx:125-135` always displays a green Wi-Fi icon.
- Failed data loads can leave empty arrays plus a banner, which is easy to
  mistake for a real empty dataset.

**Impact**

The shell opens offline, but the operator cannot reliably view current data or
perform work. This contradicts the app’s offline-first positioning.

**Required remediation**

Either implement encrypted/local snapshots and a conflict-aware write queue, or
explicitly make the product online-required with a persistent offline/read-only
state and last-sync timestamp.

# Medium findings

## M1. Bill month selection does not load that month

`useEntityStore.js:51-61` loads only the current month. `AppPage.jsx:73-92`
changes `billMonth` and generates for it, but no effect fetches bills for the
selected month. Post-generation refreshes also call `getBills` without a month.
Historical selection therefore displays current-month data.

## M2. Several versioned mutations bypass concurrency control

`src/lib/api.js:324-330`, `:628-642`, and `:1016-1029` update customer status,
bill lock state, and import status/delete without checking or incrementing
`version`. A payment can race with a lock because lock/unlock does not change the
bill version.

## M3. Import deletion is one-tap permanent deletion

`src/pages/Imports.jsx:146-160` calls delete directly; `src/lib/api.js:1024-1029`
permanently deletes the row. There is no confirmation, busy lock, soft delete,
audit trail, or undo.

## M4. Service-worker install/update can produce a partial mixed deployment

`public/sw.js:92-101` catches every cache-add failure and still completes install.
It immediately calls `skipWaiting`; activation deletes older caches and claims
clients. `public/register-sw.js` has no update-available/reload flow or
user-visible failure state. A new worker can take control without a complete new
shell.

## M5. The service worker is excluded from quality checks

`eslint.config.js:7` ignores `public/sw.js`; coverage only includes `src/**`; no
service-worker tests exist. The file owns security-sensitive routing that keeps
Supabase and API traffic out of caches.

## M6. Modal and form primitives are not accessible

`src/components/ui.jsx:52-80` lacks `role="dialog"`, `aria-modal`, heading
association, initial focus, focus trap, and focus restoration. Close buttons
have no accessible names. `Field` labels are not associated with inputs.
Required/error state is not announced. Overlay taps can discard unsaved data.

## M7. Toasts and errors are not announced

`src/components/ui.jsx:29-49`, `src/components/login.jsx:70-85`, and the shell
error banner lack suitable `role="status"`, `role="alert"`, `aria-live`, or
field associations. Fetch failures in subscription history become an empty list
and can be presented as “no history.”

## M8. Dark mode has severe contrast failures

`src/index.css:37-47` creates dark surfaces, while pages and modals frequently
hardcode `#111`, `#374151`, and white backgrounds. Primary names and bill values
can be nearly invisible in dark mode. Replace hardcoded colors with theme tokens
and automate contrast checks for both themes.

## M9. Undefined utility classes silently do nothing

Dashboard and Delivery use Tailwind-style classes such as `space-y-6`, `flex`,
`grid-cols-2`, and `text-gray-600`, but no Tailwind dependency/config exists and
`index.css` does not define them. `Delivery` also passes `className` to `Btn`,
which does not forward it. Intended layout and responsive behavior are missing.

## M10. Dashboard actions and recent-bill values are broken

- `Dashboard.jsx:84-107` supplies a StatGrid `action`, but `ui.jsx:172-193`
  ignores it; “View Log” is not interactive.
- `Dashboard.jsx:152-154` passes the amount as children to `Badge`, while
  `Badge` only reads `label`; recent bill amount badges are blank.

## M11. Mobile shell and touch behavior need hardening

The header forces title, theme, date, and sign-out into one row. Several icon and
small buttons are below a 44×44 target. There is no top safe-area padding despite
`black-translucent`, no `100dvh`/keyboard-safe modal height, no reduced-motion
rule, and limited explicit `:focus-visible` behavior.

## M12. Deployment security headers are incomplete

`index.html:13-14` contains a useful fallback CSP, but `script-src` unnecessarily
allows `'unsafe-inline'`; `object-src`, `base-uri`, and `form-action` are absent.
`frame-ancestors` cannot be enforced by a meta CSP. Production hosting must set
CSP with `frame-ancestors 'none'`, HTTP Strict Transport Security,
`X-Content-Type-Options`, `Referrer-Policy`, and `Permissions-Policy` headers.

## M13. Supabase environment configuration is not validated

`src/lib/supabaseClient.js:3-6` calls `createClient` with unchecked values. A
missing/malformed production environment variable fails during module loading,
outside the React error boundary, without an actionable setup message.

## M14. Date and currency presentation is inconsistent

Several pages display raw `YYYY-MM`/`YYYY-MM-DD`. Subscription history uses the
device timezone rather than explicit Asia/Kolkata. `getDailyInventory` at
`src/lib/api.js:1172-1176` still defaults with UTC `toISOString()` instead of
`getToday()`. Some UI strings manually build `₹` values instead of using `fmt()`.

## M15. PIN entry is visible and mislabeled as an OTP

`src/components/login.jsx:47-63` uses `type="text"` and
`autoComplete="one-time-code"` for a reusable administrative PIN. Use a password
field with numeric input, an optional show/hide control, and accessible error
association.

## M16. `callApi` is an untested high-risk god function

`src/lib/api.js` is 1,349 lines; Fallow reports `callApi` cognitive complexity
225 and high untested risk. Financial, auth, delivery, and admin actions share
one switch, making surgical changes difficult. Split by domain behind a
backward-compatible dispatcher after adding tests.

# Low and operational findings

## L1. `check-all` uses unpinned `npx` tools

Fallow, Madge, Prettier, depcheck, and cspell are not dev dependencies.
`npm run fallow` installed the current registry version dynamically. Results can
change without a lockfile change and require network access.

## L2. The configured cycle check omits JSX

`package.json:17` runs `madge --circular src/`, which processed 28 files. Running
with `--extensions js,jsx` processed 58 files and found no cycles. Update the
script and pin Madge.

## L3. Formatting is not clean or gated

`npm run cprettier` fails for `index.html`, `package.json`,
`src/components/login.jsx`, and `src/lib/api.js`. Prettier is absent from
`check-all` and CI.

## L4. Git whitespace validation fails

`git diff --check` reports trailing whitespace in
`supabase/security_functions.sql:42`.

## L5. README has no operational documentation

`README.md` contains only the project name. It does not document Node/npm
versions, environment setup, schema deployment order, release checks, backups,
restore, rollback, PWA updates, or the real authentication/RLS assumptions.

## L6. Manifest compatibility can improve

`manifest.json` lacks a dedicated raster 192×192 icon and stable manifest `id`.
Validate maskable safe zones and installation in browser tooling.

## L7. Dependency maintenance updates are available

Verified maintenance updates, not known vulnerabilities:

- `@supabase/supabase-js`: 2.110.3 → 2.110.7
- `lucide-react`: 1.24.0 → 1.25.0
- `vite`: 8.1.4 → 8.1.5

Review release notes and rerun all checks after updating.

# Positive controls already present

The app also contains useful production-oriented work that should be preserved:

- Service worker registration is now present in `index.html:27-31`.
- Supabase, `/api`, `/functions`, `/rest/v1`, and `/auth/v1` requests are kept
  network-only by `public/sw.js:123-142`.
- Same-origin asset discovery and extension filtering reduce unsafe pre-caching.
- `updateWithVersion` uses a proper compare-and-set update plus
  `.select().single()` for supported entity updates.
- The database has unique customer/date and customer/month constraints.
- Payment entry rejects non-positive amounts, overpayment, locked bills, and one
  class of version conflict.
- Daily log generation fetches customers, subscriptions, pauses, and existing
  logs in parallel.
- `useBusy` prevents duplicate submissions for modal save controls.
- Toast timers are replaced and cleaned up safely.
- `getToday()` and daily-generation weekday calculation are Asia/Kolkata-aware.
- Build sourcemaps are disabled.
- `.env` is now ignored; no service-role credential is present in browser code.
- The app has a React error boundary and a partial-load retry banner.
- Currency formatting generally uses the shared `fmt()` helper.
- npm audit reports no known dependency vulnerabilities.

# Required release plan

## Phase 0 — Stop release and protect data

1. Freeze production deployment.
2. Export and verify a Supabase backup and restore procedure.
3. Inspect live `pg_policies`, function grants, function definitions, and table
   columns; do not assume the checked-in SQL matches production.
4. Revoke anon/public destructive access immediately.

## Phase 1 — Rebuild the security boundary

1. Implement real Supabase Auth for the operator.
2. Deny anonymous table access and settings access.
3. Replace PIN RPCs with contract-tested setup/login/rotation behavior, or use
   the PIN only as a local unlock on top of real authenticated identity.
4. Add rate limits, durable lockout, forced initial credential rotation, and
   audit logging.

## Phase 2 — Make money operations transactional

1. Implement database RPCs for payment, bill generation, adjustment, and credit
   note application.
2. Put idempotency checks before mutations.
3. Lock affected rows and update source/audit/bill records in one transaction.
4. Add financial constraints and integration tests for retries, conflicts,
   locked bills, partial failures, and exact balances.

## Phase 3 — Repair normal operator workflows

Fix and test, in order:

1. Delivery toggle preservation and selected-date loading.
2. Bill amount calculation and month fetching.
3. Import total persistence.
4. Subscription add/edit form contracts.
5. Customer deactivation.
6. Adjustment/credit-note targeting and UI.
7. Form default initialization.
8. Remove or rebuild auto-rollover.

## Phase 4 — Establish a real release gate

1. Convert SQL to versioned migrations and test clean install plus upgrade.
2. Pin all quality tools and make `check-all` deterministic and green.
3. Run the full gate in CI, including SQL/RLS/RPC integration tests.
4. Add incremental coverage thresholds focused first on auth, money, delivery,
   and entity loading.
5. Add browser E2E tests for the main operator journey and PWA registration,
   update, and offline behavior.

## Phase 5 — UX, accessibility, and operations

1. Build accessible modal/form primitives and error announcements.
2. Fix dark mode, responsive layouts, touch targets, and real connectivity state.
3. Add deployment security headers.
4. Document environment setup, deployment, migrations, backups, restore,
   rollback, incident response, and PWA cache/version procedures.

# Production acceptance criteria

Do not label the app production-ready until all of the following are true:

- Anonymous users cannot read or mutate business/settings data.
- Login, lockout, rotation, expiry, logout, and direct API bypass tests pass.
- Payment/billing/adjustment/credit-note operations are transactional and
  idempotent under retries and concurrency.
- Delivery toggles preserve quantity and selected dates display correctly.
- Imports, subscriptions, deactivation, and historical billing pass integration
  and E2E tests.
- Clean install and upgrade migrations pass in CI.
- `npm run check-all`, formatting, SQL checks, and coverage thresholds pass.
- Offline behavior is either fully implemented or accurately represented as
  online-required.
- Backup/restore and rollback are documented and tested.
