## 1. CRITICAL findings

### 1.1 PIN stored as plaintext, no rate limit, 10 000-space brute force
**File:** `supabase_sql_editor` (PIN seed) · `src/lib/api.js` `verifyPIN` · `src/hooks/useAuth.js`
**Problem:** `settings.value` holds the PIN in cleartext. Brute-forcing 4 digits against a 10 000-space is trivial; no throttling, no lockout, no MFA. RLS is wide open, so an attacker with the anon key can iterate directly against Supabase.
**Evidence:** SQL seed: `INSERT INTO settings (key, value) VALUES ('PIN', '1234')`. `verifyPIN` does a single equality check.
**Fix direction:** Hash PIN with Argon2id server-side (Supabase Edge Function or DB trigger), enforce exponential backoff via `failed_attempts` column, accept ≥6 digits (OWASP guidance). Treat 4-digit PIN as a session unlock only, not as the security boundary — move real auth to RLS + JWT.

### 1.2 `eraseAllData` action callable by any anon key holder
**File:** `src/lib/api.js` `eraseAllData` (the entire wipe path)
**Problem:** The "safety" check `payload.confirm === "ERASE"` is enforced **client-side only**. Anyone with the anon key (extracted from the deployed bundle at `/assets/index-*.js`) can `POST` directly to Supabase and wipe every table. The `customers`, `bills`, `daily_logs`, etc. RLS policies are all `USING (true)`.
**Evidence:** `if (payload.confirm !== "ERASE") throw…` runs after the switch has already received the request. The check is UI sugar.
**Fix direction:** Gate destructive ops behind a Postgres `SECURITY DEFINER` RPC that requires the caller's JWT role. RLS must reject writes from the `anon` role on these tables; only `authenticated` or a service role can do destructive work.

### 1.3 `.env` committed to the repo
**File:** `.env` (project root) — also see `src/lib/supabaseClient.js` (no guard for missing env)
**Problem:** Supabase URL + anon JWT are committed. Anon key is *meant* to be public, but the act of committing a `.env` trains the next contributor to do the same. There is no `.env.example` template. `supabaseClient.js` silently constructs `createClient(undefined, undefined)` on missing env (no throw) — the failure mode is "everything breaks silently".
**Evidence:** `.gitignore` does not list `.env`. Repo search shows live credentials.
**Fix direction:** Add `.env` to `.gitignore` (it isn't currently). Add `.env.example` with placeholders. Rotate the anon key once. Add a guard in `supabaseClient.js`: `if (!supabaseUrl || !supabaseAnonKey) throw new Error(...)`. Consider an `import.meta.env` typecheck at build time.

### 1.4 RLS is wide open on every table
**File:** `supabase_sql_editor` — every `CREATE POLICY` is `FOR ALL USING (true) WITH CHECK (true)`
**Problem:** Combined with 1.1, 1.2, and the public anon key, the entire DB is readable and writable by anyone who reaches the Supabase URL. There is zero authorization layer server-side. The PIN "auth" in the app is purely a UI gate.
**Evidence:** The schema enables RLS but writes `USING (true)` policies, which is the same as disabling it for practical purposes.
**Fix direction:** Define at minimum: `anon` role = no read/write, `authenticated` role = RLS by `customer_owner_id` (you don't have this column today — add it), `service_role` = full. For an internal single-operator tool, consider dropping the anon-key-as-auth model entirely and putting a Cloudflare Worker in front with HTTP Basic or session cookies.

## 2. HIGH findings

### 2.1 Race conditions in money paths
**Files:** `src/lib/api.js` — `recordPayment`, `generateMonthBill`, `applyAdjustment`
**Problem:** All three read a row, compute a new value, then update. No `SELECT ... FOR UPDATE`, no atomic SQL, no transaction. Two parallel updates to the same bill will lose one of the writes (last-write-wins, silently).
- `recordPayment`: reads bill, computes `newPaid = amount_paid + amount`, updates. Classic lost-update.
- `generateMonthBill`: computes amount, then either insert or `update` with manual `version + 1` (bypasses `updateWithVersion`).
- `applyAdjustment`: two-step update (bill amount, then adjustment.applied). If step 1 succeeds and step 2 fails, bill is mutated but adjustment is not marked applied.
**Fix direction:** Wrap each in a Postgres RPC that does the read+write atomically: `UPDATE bills SET amount_paid = amount_paid + $1 WHERE id = $2 RETURNING amount_paid, amount`. For adjustments, do both writes in a single transaction or a single RPC.

### 2.2 Session token is unauthenticated
**File:** `src/lib/api.js` `verifyPIN` · `src/hooks/useAuth.js`
**Problem:** Token = `crypto.randomUUID()` generated **client-side** and stored in `sessionStorage`. Subsequent calls to `callApi` never include it as a header — Supabase has no way to know who's calling. `sessionSecret` is also written but never read anywhere. The token is theatre.
**Evidence:** `verifyPIN` returns `{ token }` and writes it. No `Authorization` header is ever constructed in `callApi` (Supabase uses its anon key from `createClient` for all calls).
**Fix direction:** Either (a) make the token meaningful: have `verifyPIN` return a Supabase-issued JWT and switch the client to email-link/OTP auth; or (b) drop the token concept and rely on server-side RLS + a single cookie. Don't keep a token that "looks" secure but isn't.

### 2.3 Subscription `milkType` enum is broken
**Files:** `src/components/modals/SubscriptionModal.jsx` · `src/lib/api.js` `saveSubscription` · `src/lib/constants.js`
**Problem:** `SubscriptionModal` hardcodes the dropdown to `["FULL_CREAM", "SKIMMED", "BUFFALO"]` (all UPPER_SNAKE). The schema default is `"Full Cream"`, and `constants.js` exports `MILK_TYPES = ["Full Cream", "Toned", ...]`. The form sends `milk_type: "FULL_CREAM"` to a column that the rest of the system compares to `"Full Cream"`. The mismatch will silently break `generateDailyLogsForDate` (which writes the subscription's `milk_type` to `daily_logs.product`) and any future `WHERE product = 'Full Cream'` queries.
**Evidence:** `SubscriptionModal` `value="FULL_CREAM"`, schema default `"Full Cream"`.
**Fix direction:** Pick one source of truth. Either (a) change the form to use `constants.MILK_TYPES` values, or (b) add `FULL_CREAM`/`SKIMMED`/`BUFFALO` to the canonical list. Don't let the form and constants drift.

### 2.4 `SubscriptionsListModal` displays "Unknown Customer" for every row
**File:** `src/components/modals/SubscriptionsListModal.jsx` line 50: `sub.customerName || "Unknown Customer"`
**Problem:** The `Subscription` shape from `mapSubscriptionFromApi` is `{ id, customerId, milkType, qty, deliveryDays, isActive, version }`. It does **not** carry `customerName`. Every subscription row shows "Unknown Customer" until you join client-side. Same bug in `SubscriptionHistoryModal` title.
**Evidence:** `mapSubscriptionFromApi` doesn't expose name; the modal assumes it's there.
**Fix direction:** Add a `customerName` projection either in the API mapper (if you add a join) or resolve it client-side from `customers` prop (e.g. `customers.find(c => c.id === sub.customerId)?.name`).

### 2.5 `idempotencyKey` is dead code
**Files:** every `callApi` insert path (`addCustomer`, `addMilkImport`, `addAdjustment`, `addCreditNote`, `addPausePeriod`, `addMilkBrand`, `addAdHocLog`, `generateDailyLogsForDate`, `toggleDeliveryLog`, `recordPayment`, `saveSubscription`, …)
**Problem:** Most insert/update payloads include `idempotencyKey: <string>`. The backend `supabase.from(...).insert([...])` calls ignore the key. `toggleDeliveryLog` even uses `Date.now().toString()` which collides in same-millisecond clicks. So the field is decorative and gives a false sense of safety.
**Fix direction:** Either (a) add a unique constraint on an `idempotency_key` column and use `onConflict: "idempotency_key"` for upsert, or (b) remove the field. Don't ship keys that aren't enforced.

### 2.6 `SubscriptionHistoryModal` refires on every parent render
**File:** `src/components/modals/SubscriptionHistoryModal.jsx` line 12: `useEffect(() => { ... }, [data, handlers])`
**Problem:** `handlers` is `{ ...state, ...handlers, auth }` constructed in `App.jsx` on every render. The effect refetches history every time anything in the app re-renders. With 4–5 modals open in a session, this hammers the API.
**Fix direction:** Stabilize the handlers object (memoize in `App.jsx`) OR depend only on `data?.id`.

### 2.7 Service worker registration silently fails
**File:** `public/register-sw.js` — `.catch(err => console.error(...))`
**Problem:** If SW registration fails, the user has no offline support and no diagnostic. On the operator's flaky phone network, this is the failure mode they'll hit most.
**Fix direction:** Add a UI banner on persistent failure (e.g. on `navigator.serviceWorker.ready` rejecting, or after a 5s timeout without activation).

## 3. MEDIUM findings

### 3.1 `useAppState.ui.setF` memo is wasted
**File:** `src/hooks/useAppState.js` `useAppUi` → `setF`
**Problem:** `setF = useCallback((k) => (e) => setForm(...), [])` — the inner function is fresh on every call. Memo doesn't help. Minor perf, but in a list of 100+ customers with `onChange={onChange("name")}` per row, the inline arrow is reallocated constantly.
**Fix direction:** Use a `useRef` cache keyed by `k`, or have callers use `onChange={(e) => setForm(...)}` and stop pretending it's memoized.

### 3.2 `useEntityStore.safeFetch` returns `null` on error
**File:** `src/hooks/useEntityStore.js`
**Problem:** On failure, the consumer checks `if (custs !== null)` then `setCustomers([])`. But the user has no way to know *which* entity failed except by reading the `loadErrors` banner. Also, `refresh()` clears errors and bumps a key — but doesn't tell you why it failed.
**Fix direction:** Keep the banner (good) but also toast a single summary toast on partial-load, and keep stale data instead of overwriting with `[]` on a single failed refetch.

### 3.3 `today` is memoized once and never auto-advances
**File:** `src/hooks/useAppState.js` `clampedToday()` + `useMemo([], [])`
**Problem:** If the operator keeps the app open past midnight, `today`, `logDate`, and `billMonth` are frozen. A 24/7 admin tool on a phone is exactly the case where this bites. `todayLogs` in `useAppDerived` is also pinned to "yesterday" after midnight until reload.
**Fix direction:** Re-derive `today` from `useEffect` on `visibilitychange`/`focus`, or run a `setInterval` at 60s that bumps a `tick` state. Don't tie `today` to a module-level Date constant.

### 3.4 `getDay()` uses local time, not Asia/Kolkata
**File:** `src/lib/api.js` `generateDailyLogsForDate` line ~210
**Problem:** `new Date(\`${date}T00:00:00\`).getDay()` parses the date as **local** time, not Asia/Kolkata. An operator in any other timezone will get the wrong weekday, which means the wrong customers get deliveries generated. Combined with 3.3, this is a real daily-delivery bug waiting to happen.
**Fix direction:** Either pass the date as a `YYYY-MM-DD` string to a Postgres function that handles timezone, or compute weekday with `Intl.DateTimeFormat('en-US', { weekday: 'short', timeZone: 'Asia/Kolkata' })`.

### 3.5 `recordPayment` lets users overpay silently
**File:** `src/lib/api.js` `recordPayment` · `src/components/modals/PaymentModal.jsx`
**Problem:** Default amount = `data.amount - data.paid` (full pending). User can enter any positive number. If they enter `1.5x` the pending, `newPaid` exceeds `amount`, status becomes `"Paid"` and the extra is just... lost. No refund record, no overpay warning.
**Fix direction:** Either cap the input at pending, or create a `credit_notes` entry for the overpay and surface it. Money should not silently disappear.

### 3.6 XSS surface: customer-controlled strings rendered into HTML
**Files:** many — `name`, `address`, `reason`, `note` rendered via JSX in `pages/*.jsx`, `components/forms.jsx`, `components/ErrorBoundary.jsx`
**Problem:** React JSX auto-escapes string children, so direct `{c.name}` is safe. But several places use `style={{...}}` with template strings containing user data (e.g. the WhatsApp button `title`). Also: `whatsapp` builds `wa.me/...` URLs and opens them via `window.open` — if the phone number contains a malicious payload (e.g. from a CSV import), the URL is malformed but not exploitable. Mostly safe today, fragile if any developer switches to `dangerouslySetInnerHTML` later.
**Fix direction:** Add an ESLint rule `react/no-danger` (already standard in `eslint-plugin-react`). Document the policy. Audit `useAppDerived` for any field concatenation that escapes JSX.

### 3.7 `Login` input is `type="text"` for a PIN
**File:** `src/components/login.jsx`
**Problem:** `type="text"` means the value is logged, autofilled, visible to screen recorders. For a 4-digit PIN, `type="password"` (or `type="tel"` with `inputMode="numeric"`) is correct.
**Fix direction:** Change to `type="password"` with `inputMode="numeric"`. Drop the misleading `autoComplete="one-time-code"` (that's for SMS OTP, not app PIN).

### 3.8 `Modal` Escape handler ignores IME composition
**File:** `src/components/ui.jsx` `Modal` useEffect
**Problem:** Pressing Escape during an IME composition (Chinese, Japanese, Hindi transliteration) closes the modal and discards the input.
**Fix direction:** Check `e.isComposing` (or `e.keyCode === 229`) before closing.

### 3.9 `More.jsx` "Diagnostics" panel shows hardcoded fake data
**File:** `src/pages/More.jsx` — the `DIAGNOSTICS` array
**Problem:** Items like "SystemState rows: 512 — high", "PINSalt configured: OK", "Milk import sheets: Present" are **hardcoded constants** rendered to look like live diagnostics. The user trusts these. After one real `runDiagnostics` call, only the status is updated; the row values are still fake.
**Fix direction:** Either run a real check per row (10 RPCs is fine) or relabel the panel "Demo Diagnostics" so the operator isn't fooled.

### 3.10 `getDailyInventory.remaining` is unclamped
**File:** `src/lib/api.js` `getDailyInventory`
**Problem:** If deliveries exceed imports (e.g. opening stock carried over, or ad-hoc logs), `remaining` goes negative. No "in stock from yesterday" tracking.
**Fix direction:** Either clamp to `Math.max(0, remaining)` and add a "carried over" line, or expose a rolling stock model.

### 3.11 `updateLogEntry` is not scoped to a customer
**File:** `src/lib/api.js` `updateLogEntry`
**Problem:** `eq("id", payload.logId)` only — anyone with a logId can flip any log. Cross-customer manipulation is possible in the data layer (currently the UI guards, but `callApi` is a public surface in this app).
**Fix direction:** Add `.eq("customer_id", payload.customerId)` if the caller has it, or scope by a session token from `useAuth`.

### 3.12 Inconsistent mapping layer
**File:** `src/lib/api.js` vs. `src/hooks/handlers/`
**Problem:** `mapCustomerToApi`, `mapImportToApi`, `mapPaymentToApi` exist, but `mapSubscriptionToApi`, `mapAdjustmentToApi`, `mapCreditNoteToApi` do not. The handlers construct payloads inline, which means the convention documented in AGENTS.md is broken in 3 of 9 entities.
**Fix direction:** Add the missing `mapXToApi` functions and route through them. New code should follow the pattern, not perpetuate the drift.

### 3.13 `Dashboard` quick action has no busy state
**File:** `src/pages/Dashboard.jsx` → "Generate Bills"
**Problem:** `onGenerateBill` runs `Promise.allSettled` over every active customer. Could be 10s+. The button is not disabled, no spinner, user can double-click.
**Fix direction:** Wrap the quick-action button in `useBusy` like Delivery does. Or render the existing button as `disabled` during the run.

### 3.14 `Modal` close-on-overlay-click is awkward on mobile
**File:** `src/components/ui.jsx` `Modal`
**Problem:** Tapping the overlay closes the modal. On mobile, a long-tap that drifts to the overlay drops the user's input. No save-on-close, no confirmation.
**Fix direction:** Trap focus inside the modal, and use a dedicated Close button. Add a confirm-on-unsaved-changes prompt if `form` is dirty.

### 3.15 `more.jsx` `resolveDiagnostics` re-computes on every render
**File:** `src/pages/More.jsx`
**Problem:** `resolveDiagnostics(activeBrandsCount, brands)` runs every render. Tiny perf, easy fix.
**Fix direction:** `useMemo` it.

### 3.16 `useAppDerived` re-creates arrays on every dep change
**File:** `src/hooks/useAppDerived.js`
**Problem:** Memo dependency list is `customers, imports, bills, brands, logs, custSearch, custFilter, impFilter, billFilter` — any change recomputes everything. If `logs` changes (e.g. a delivery flip), `filteredC` is recomputed. Not wrong, just wasteful.
**Fix direction:** Split into two memos: list-derived (`filteredC`, `filteredImports`, `filteredB`) and metric-derived (`activeC`, `totalRevenue`, `pendingDues`).

### 3.17 `App.jsx` `ctx` is a new object every render
**File:** `src/App.jsx`
**Problem:** `const ctx = { ...state, ...handlers, auth }` — fresh ref every render. All modals re-render on any state change. Combined with 2.6, the history modal refetches needlessly.
**Fix direction:** `useMemo` the ctx.

### 3.18 `BulkUpsertLogs` does not guard against empty input
**File:** `src/lib/api.js` `bulkUpsertLogs`
**Problem:** `payload.logs.map(...)` will throw on `undefined`. A misconfigured caller (e.g. `handlers.bulkUpsertLogs()` with no args) crashes the action.
**Fix direction:** Guard with `if (!Array.isArray(payload.logs)) return;`.

### 3.19 `getBillText` does not use `fmt`
**File:** `src/lib/api.js` `getBillText`
**Problem:** Builds a string with `\u20B9${toNum(bill.amount).toFixed(2)}` — duplicates `fmt()`'s logic and skips the en-IN locale formatting (`1,23,456.78` becomes `123456.78`).
**Fix direction:** Import `fmt` from `utils.js` and use it.

### 3.20 `StatGrid` accepts both `items` and `stats`
**File:** `src/components/ui.jsx` `StatGrid`
**Problem:** Two props for the same thing. `items || stats || []`. Two callers use `items` (`Delivery.jsx`, `Imports.jsx`), one uses `stats` (`Dashboard.jsx`). Drift in naming convention.
**Fix direction:** Pick one. `items` is the better name; update `Dashboard.jsx`.

### 3.21 Inline style thrash
**Files:** all pages, all modals
**Problem:** Every render reallocates dozens of style objects. Not catastrophic on a 6-tab admin tool, but it makes diffs noisy and theme support (dark mode) patchy. AppShell has `style={{...}}` with hardcoded colors that ignore the `--brand-*` CSS variables.
**Fix direction:** Decide: inline styles for everything → keep, or move to CSS modules. Right now the project is in a halfway state where some styles are in `index.css` and some are inline.

## 4. LOW / style

- **Unused state**: `useAppState.ui.modal` vs `useAppState.ui.form` are kept in sync manually (`openModal` sets both). One source of truth (just `modal: { type, data, form }`) would be simpler.
- **`useAppState` recomputes `today` with `useMemo([], [])`** — see 3.3; also the same `clampedToday` is called from inside `useAppUi` and again from `useAppState`.
- **Test file naming**: `pages/Delivery.test.jsx` exists but `pages/Dashboard.test.jsx`, `pages/Customers.test.jsx`, etc. do not. There's a comment in the test about the `mapXFromApi` mocks being missing — that's a fragile pattern.
- **Build artifacts**: `vite.config.js` sets `sourcemap: "hidden"` — good, but `.fallow` and `.tmp` directories exist with no gitignore entries. Add them.
- **Manifest `theme_color`**: hardcoded `#1e40af`. CSS uses `--brand-600: #2563eb`. The two will drift visually.
- **Hardcoded months in `Dashboard.jsx`**: `MONTH_NAMES = ["Jan", ...]`. Move to a `constants` or `utils` helper for reuse.
- **`cspell.json` words list is short** — add domain words (e.g. `idempotency`, `PGRST`, `Madge`, `Fallow`, `adj`, `custId`).
- **`SubscriptionModal` form uses `quantity` but the data field is `qty`** — silent mismatch; `mapSubscriptionToApi` (if it existed) would catch this.
- **`generateMonthBill` is non-deterministic** for a customer with no `ratePerLiter` (always 32). Document the fallback or surface it.

Missing coverage for **business-critical** paths:
